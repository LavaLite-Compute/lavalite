# LavaLite Job File Format

## Overview

A LavaLite *job file* is a shell script generated by the scheduler
(MBD) and executed by the execution daemon (SBD) to start a batch job.

The job file is a transport wrapper, not an interactive shell
environment and not a replica of a user login session.

Primary goals:

- correctness
- determinism
- POSIX portability
- reproducibility

Interactive shell conveniences are explicitly out of scope.

---

## Interpreter Contract

- The job file interpreter is `/bin/sh`
- The script must be valid POSIX sh
- No assumptions are made about:
  - bash extensions
  - user dotfiles
  - login or interactive shell initialization

The value of `$SHELL` in the exported environment does not
change the interpreter used to execute the job file.

---

## File Structure

A LavaLite job file has the following logical structure:

    #!/bin/sh
    Environment block
    Execution gate
    User command
    Exit handling

Each section has strict behavioral meaning.

---

## Environment Block

The environment block serializes selected environment variables from
the submitting process.

Markers:

    # LSBATCH: Environments
    ...
    # LSBATCH: ENVEND

These markers are informational and allow debugging and inspection.
They are not interpreted by the shell.

---

### Environment Serialization Rules

Environment variables are emitted according to the following contract.

#### Variable Name

A variable name must match the POSIX shell identifier grammar:

    [A-Za-z_][A-Za-z0-9_]*

Invalid identifiers are silently skipped.

Examples:

    PATH                valid
    VAR123              valid
    _FOO                valid
    1VAR                invalid
    LD_LIBRARY-PATH     invalid
    BASH_FUNC_ml%%      invalid

Exported bash function names are explicitly ignored.

---

#### Variable Filtering

Variables are skipped if:

- the name is not a valid POSIX identifier
- the variable represents an exported bash function
- the variable belongs to an internal deny list
- the value contains literal newline characters

Filtering rules may differ slightly for interactive jobs.

---

#### Variable Value

Values are treated strictly as data, never behavior.

Rules:

- values may contain arbitrary characters
- values must not contain newline characters
- values are emitted as a single quoted shell word

Emission format:

    NAME='value'; export NAME

---

## Execution Gate

Before executing user commands, SBD waits for the job
execution gate file:

    go="$LSB_JOBDIR/go"
    while [ ! -f "$go" ]; do
        sleep 1
    done

The job does not begin execution until the gate appears.

---

## User Command Section

User commands follow the marker:

    # LSBATCH: User input

The command is inserted verbatim as provided during submission.

Example:

    # LSBATCH: User input
    sleep 3600

No shell rewriting or interpretation is performed by LavaLite.

---

## Exit Handling

After command execution, the job file records completion
status atomically.

Example tail:

    ExitStat=$?
    timestamp=$(date +%s)

    out="$LSB_JOBDIR/exit"
    tmp="$LSB_JOBDIR/.exit.$$"

    (umask 077; echo "$ExitStat $timestamp" > "$tmp") && \
        mv -f "$tmp" "$out"

    rm -f "$tmp" 2>/dev/null || true
    exit $ExitStat

### Exit Contract

- Exit status is captured immediately after user command completion
- Exit information is written atomically
- File permissions are restricted (umask 077)
- The exit file becomes the authoritative completion signal
- The script exits with the user command exit code

---

## Determinism Guarantees

The LavaLite job file guarantees:

- identical execution semantics across hosts
- independence from interactive shell configuration
- reproducible environment reconstruction
- restart-safe execution coordination
- atomic completion reporting

---

## Non-Goals

The job file intentionally does not provide:

- login shell semantics
- user dotfile sourcing
- shell feature emulation
- environment mutation heuristics

Any required runtime setup must be explicitly defined
by the submitted command or external tooling.

---

## Summary

A LavaLite job file is a minimal, deterministic execution wrapper:

- POSIX `/bin/sh`
- strictly serialized environment
- scheduler-controlled execution gate
- atomic exit reporting

The format prioritizes correctness and recoverability
over convenience.
## Example Job File

The following example illustrates a typical LavaLite-generated job file.
Actual environments may contain many more variables.

    #! /bin/sh

    # LSBATCH: Environments
    PATH='/usr/bin:/bin'; export PATH
    LANG='en_US.UTF-8'; export LANG
    LAVALITE_HOME='/opt/lavalite-0.1.0'; export LAVALITE_HOME
    # LSBATCH: ENVEND

    go="$LSB_JOBDIR/go"
    while [ ! -f "$go" ]; do
        sleep 1
    done

    # LSBATCH: User input
    sleep 3600

    ExitStat=$?
    timestamp=$(date +%s)

    out="$LSB_JOBDIR/exit"
    tmp="$LSB_JOBDIR/.exit.$$"

    (umask 077; echo "$ExitStat $timestamp" > "$tmp") && \
        mv -f "$tmp" "$out"

    rm -f "$tmp" 2>/dev/null || true
    exit $ExitStat


---

## Environment Exclusion Rules

The environment serialized into the job file is derived from the
submission environment (`environ`) but filtered strictly before
emission.

Variables are **excluded** when any of the following conditions apply.

### 1. Invalid Shell Identifier

Environment variable names must satisfy the POSIX shell identifier rule:

    [A-Za-z_][A-Za-z0-9_]*

Variables failing this rule are skipped.

Examples excluded:

    1VAR
    LD_LIBRARY-PATH
    VAR-NAME
    NAME.WITH.DOT

---

### 2. Exported Bash Functions

Bash exported functions are explicitly ignored.

Typical examples:

    BASH_FUNC_ml%%
    BASH_FUNC_module%%

These entries are shell implementation artifacts and are not portable.

---

### 3. Internal Deny List

Certain variables are intentionally excluded by LavaLite internal policy
(`env_is_denied()`).

These typically include:

- scheduler internal variables
- execution-control variables
- environment entries that would corrupt deterministic execution
- variables reconstructed independently on execution hosts

The exact deny list may differ between interactive and batch jobs.

---

### 4. Multiline Values

Variables whose values contain literal newline characters are excluded.

Multiline values cannot be safely represented as a single POSIX shell
assignment and would break deterministic reconstruction.

---

### 5. Non-Representable Data

Environment entries that cannot be safely emitted using the canonical
format:

    NAME='value'; export NAME

are skipped rather than transformed.

LavaLite prefers omission over heuristic rewriting.

---

## Serialization Principle

Environment reconstruction follows a strict rule:

    Only portable, deterministic, single-line data is transported.

Anything ambiguous, shell-dependent, or non-portable is excluded.

This guarantees that job execution behavior depends only on explicitly
representable state and not on submit-host shell implementation details.
